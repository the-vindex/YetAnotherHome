## 1. Brigadier command patterns in modern mods

### Structure: static `register(CommandDispatcher)` per command-domain class

FTB Essentials (1.20.1/main) uses one class per command domain, each with a static
`register(CommandDispatcher<CommandSourceStack>)`, and a single aggregator:

```java
// dev/ftb/mods/ftbessentials/command/FTBEssentialsCommands.java
public static void registerCommands(CommandDispatcher<CommandSourceStack> dispatcher) {
    HomeCommands.register(dispatcher);
    WarpCommands.register(dispatcher);
    TeleportCommands.register(dispatcher);
    TPACommands.register(dispatcher);
    ...
}
```
Source: https://github.com/FTBTeam/FTB-Essentials/blob/1.20.1/main/common/src/main/java/dev/ftb/mods/ftbessentials/command/FTBEssentialsCommands.java

Command classes keep the brigadier tree thin and delegate to plain static methods
(`home(player, name)`, `setHome(player, name)`, ...) that return the int result:

```java
// HomeCommands.java (FTB Essentials 1.20.1)
dispatcher.register(Commands.literal("home")
    .requires(FTBEConfig.HOME)                       // predicate on CommandSourceStack
    .executes(ctx -> home(ctx.getSource().getPlayerOrException(), "home"))
    .then(Commands.argument("name", StringArgumentType.greedyString())
        .suggests((ctx, builder) -> SharedSuggestionProvider.suggest(getHomeSuggestions(ctx), builder))
        .executes(ctx -> home(ctx.getSource().getPlayerOrException(),
                              StringArgumentType.getString(ctx, "name")))));
```
Source: https://github.com/FTBTeam/FTB-Essentials/blob/1.20.1/main/common/src/main/java/dev/ftb/mods/ftbessentials/command/HomeCommands.java

Essential Commands goes further: one class per command implementing a shared interface,
registered by a central `EssentialCommandRegistry` (packages `commands/`, `commands/suggestions/`).
For 9 commands, FTB's "one class per domain" granularity is the better fit.

### No raw-type casts needed

The raw `(LiteralArgumentBuilder)` casts in the decompiled code are decompiler artifacts.
Exact 1.20.1 signatures [javap `net.minecraft.commands.Commands`]:

```
public static LiteralArgumentBuilder<CommandSourceStack> literal(String);
public static <T> RequiredArgumentBuilder<CommandSourceStack, T> argument(String, ArgumentType<T>);
```

Chaining `.requires(...).then(...).executes(...)` on these keeps the full generic type;
written-by-hand code never needs a cast.

### Suggestions

[javap `net.minecraft.commands.SharedSuggestionProvider`], 1.20.1:

```
public static CompletableFuture<Suggestions> suggest(Iterable<String>, SuggestionsBuilder);
public static CompletableFuture<Suggestions> suggest(Stream<String>, SuggestionsBuilder);
public static CompletableFuture<Suggestions> suggest(String[], SuggestionsBuilder);
public static <T> CompletableFuture<Suggestions> suggest(Iterable<T>, SuggestionsBuilder,
        Function<T,String>, Function<T,Message>);
```

Idiom (FTB): `.suggests((ctx, b) -> SharedSuggestionProvider.suggest(names, b))` — handles the
"remaining input" prefix filtering that the hand-rolled `builder.suggest(...)` loop in the
current mod does not (it suggests all homes regardless of what was typed).

### sendSuccess / sendFailure — exact 1.20.1 signatures

[javap `net.minecraft.commands.CommandSourceStack`]:

```
public void sendSuccess(java.util.function.Supplier<Component>, boolean);   // YES: Supplier in 1.20.1
public void sendFailure(Component);
public void sendSystemMessage(Component);
public boolean hasPermission(int);
public ServerPlayer getPlayerOrException() throws CommandSyntaxException;
```

`sendSuccess` switched from `Component` to `Supplier<Component>` in 1.19.4; 1.20.1 has only the
Supplier overload. Vanilla's own `TeleportCommand` calls
`sendSuccess(Supplier, boolean)` [javap -c `net.minecraft.server.commands.TeleportCommand`].

Why prefer them over `player.sendSystemMessage`:
- `sendSuccess(supplier, false)` respects the `sendCommandFeedback` gamerule and the source's
  silent flag; the supplier means the Component isn't even built when feedback is off.
- `sendSuccess(supplier, true)` additionally broadcasts to ops (admin log) — right choice for
  admin actions like `/tphere`.
- `sendFailure` renders red and rings through to command blocks/console/RCON, not just players.
- Both work when the source is the console; `getPlayerOrException()` + `sendSystemMessage`
  silently ties feedback to a player entity only.

### RegisterCommandsEvent (Forge bus)

[forge src `net/minecraftforge/event/RegisterCommandsEvent.java`]: fired every time
`ReloadableServerResources` is rebuilt (server start **and** `/reload`); exposes
`getDispatcher()`, `getCommandSelection()` (`Commands.CommandSelection`), and
`getBuildContext()` (`CommandBuildContext`, needed for registry-aware arguments —
not needed for string/entity args). Because it re-fires on `/reload`, handlers must be
idempotent (pure `dispatcher.register` calls are).

## 2. Localization for server-side mods

### The core caveat, per Forge docs

Forge 1.20.x i18n docs (https://docs.minecraftforge.net/en/1.20.x/concepts/internationalization/):

> "A common issue is having the server localize for clients. The server can only localize in its
> own locale, which does not necessarily match the locale of connected clients."

`Component.translatable(...)` sent over the wire is resolved **on the client**. A vanilla client
(no mod jar, no lang file) renders the raw key (or the fallback arg, which doesn't exist pre-1.19.4
format). So a server-only mod must not send bare translatable components.

### What Forge gives you in 1.20.1

1. **Server-side lang loading**: `net.minecraftforge.server.LanguageHook` [forge src] loads
   `assets/<namespace>/lang/en_us.json` for **every mod/datapack namespace into the dedicated
   server's language table** (`loadLanguagesOnServer` iterates all resource namespaces for
   `"en_us"`; `loadForgeAndMCLangs` loads mc+forge). Consequence: on a Forge server,
   `Language.getInstance().getOrDefault(key)` resolves keys from your mod's `en_us.json`.
   [javap `net.minecraft.locale.Language`]: `getInstance()`, `getOrDefault(String)`,
   `getOrDefault(String,String)`, `has(String)`.

2. **`TextComponentHelper`** — purpose-built for this exact problem
   [forge src `net/minecraftforge/server/command/TextComponentHelper.java`]:

```java
/** Detects when sending to a vanilla client and falls back to sending english,
 *  since they don't have the lang data necessary to translate on the client. */
public static MutableComponent createComponentTranslation(CommandSource source,
        final String translation, final Object... args) {
    if (isVanillaClient(source))
        return Component.literal(String.format(Locale.ENGLISH,
                Language.getInstance().getOrDefault(translation), args));
    return Component.translatable(translation, args);
}
// isVanillaClient: source instanceof ServerPlayer
//   && NetworkHooks.getConnectionType(player.connection.connection) == ConnectionType.VANILLA
```

   (Forge docs i18n page explicitly recommends it: localization "is done eagerly if the receiver
   is a vanilla client", lazily otherwise.)

### What real server-side mods actually do

- **FTB Essentials (1.20.1)**: sends plain `Component.literal("Home set!")` etc. everywhere —
  no localization at all. Simple, robust with vanilla clients.
- **Essential Commands (Fabric)**: full server-side translation system — lang JSONs shipped under
  `data/essential_commands/lang/*.json`, resolved **server-side per player locale** via the
  Server Translations API (`xyz.nucleoid.server.translations.api.LocalizationTarget`); see
  `text/ECText.java`. The Fabric ecosystem has a library for this; Forge 1.20.1 does not.

**Recommendation for this mod**: keep an `en_us.json` with keys, resolve through
`TextComponentHelper.createComponentTranslation(player, key, args)` (best of both: modded clients
translate locally, vanilla clients get English), or stay with literals in one central Messages
class. Do not send raw `Component.translatable` to unknown clients.

Style: use `ChatFormatting` enum, not `TextColor.fromRgb(magic)`. Current magic numbers map to:
`5635925 = 0x55FF55 = ChatFormatting.GREEN`, `16733525 = 0xFF5555 = ChatFormatting.RED`,
`16755200 = 0xFFAA00 = ChatFormatting.GOLD`, `16711680/65280` are non-palette pure red/green.
`Component.literal("...").withStyle(ChatFormatting.GREEN)` is the idiom.

## 3. Position storage & cross-dimension teleport

### GlobalPos in 1.20.1

[javap `net.minecraft.core.GlobalPos`]:

```
public final class GlobalPos {
  public static final com.mojang.serialization.Codec<GlobalPos> CODEC;
  public static GlobalPos of(ResourceKey<Level>, BlockPos);
  public ResourceKey<Level> dimension();
  public BlockPos pos();
}
```

GlobalPos = dimension + **BlockPos** only. For homes you want exact standing position + look
angles, so define your own record; the dimension codec exists as
`Level.RESOURCE_KEY_CODEC : Codec<ResourceKey<Level>>` [javap `net.minecraft.world.level.Level`].

```java
public record StoredPos(ResourceKey<Level> dimension, Vec3 pos, float yRot, float xRot) {
    public static final Codec<StoredPos> CODEC = RecordCodecBuilder.create(i -> i.group(
        Level.RESOURCE_KEY_CODEC.fieldOf("dimension").forGetter(StoredPos::dimension),
        Vec3.CODEC.fieldOf("pos").forGetter(StoredPos::pos),
        Codec.FLOAT.fieldOf("yRot").forGetter(StoredPos::yRot),
        Codec.FLOAT.fieldOf("xRot").forGetter(StoredPos::xRot)).apply(i, StoredPos::new));
}
```
(`Vec3.CODEC` exists in 1.20.1 as `Codec<Vec3>` — list-of-3-doubles.)

Reference implementation: FTB Essentials `util/TeleportPos.java` stores
`ResourceKey<Level> dimension` (as string `dim`), `BlockPos`, nullable `Float yRot/xRot`, and a
`long time` (used for /back history), writes them with manual NBT:
https://github.com/FTBTeam/FTB-Essentials/blob/1.20.1/main/common/src/main/java/dev/ftb/mods/ftbessentials/util/TeleportPos.java

### Cross-dimension teleport — correct API for 1.20.1

[javap `net.minecraft.server.level.ServerPlayer`]:

```
public void teleportTo(ServerLevel, double, double, double, float, float);
public boolean teleportTo(ServerLevel, double, double, double, Set<RelativeMovement>, float, float); // inherited, Entity
public Entity changeDimension(ServerLevel, ITeleporter);   // portal-style; NOT for commands
```

- `ServerPlayer.teleportTo(ServerLevel, x, y, z, yRot, xRot)` **is dimension-aware**: bytecode
  shows a same-level fast path (`connection.teleport(...)`) and an else-branch that fires
  `ForgeHooks.onTravelToDimension` (cancellable `EntityTravelToDimensionEvent`), sends the respawn
  packet, and moves the player via the PlayerList machinery [javap -c ServerPlayer]. This is the
  method FTB Essentials uses for every home/warp/tpa teleport:
  `player.teleportTo(level, pos.getX() + 0.5D, pos.getY() + 0.1D, pos.getZ() + 0.5D, yrot, xrot);`
- The `Set<RelativeMovement>` overload (what vanilla `/tp` uses via
  `TeleportCommand.performTeleport`) additionally registers a
  `TicketType.POST_TELEPORT` region ticket so the destination chunk is loaded/ticks
  [javap -c `net.minecraft.world.entity.Entity` shows
  `ServerChunkCache.addRegionTicket(TicketType.POST_TELEPORT, ChunkPos, int, Object)`], and
  returns `boolean`. Prefer it (with `EnumSet.noneOf(RelativeMovement.class)`) when teleporting
  into possibly-unloaded chunks.
- Vanilla `/tp` also fires `ForgeEventFactory.onEntityTeleportCommand` →
  `EntityTeleportEvent.TeleportCommand(Entity, double, double, double)` [javap -c TeleportCommand;
  javap EntityTeleportEvent]. A well-behaved teleport mod fires this event too (FTB Essentials
  *listens* to it for /back history), letting protection/anti-cheat mods veto or adjust.

Pattern:

```java
ServerLevel level = server.getLevel(storedPos.dimension());
if (level == null) { /* dimension gone — fail gracefully */ }
var event = ForgeEventFactory.onEntityTeleportCommand(player, x, y, z);
if (!event.isCanceled())
    player.teleportTo(level, event.getTargetX(), event.getTargetY(), event.getTargetZ(), yRot, xRot);
```

Current mod's bug: `/home` teleports to stored x/y/z **in whatever dimension the player is
currently in** (`(ServerLevel) player.getCommandSenderWorld()`), because dimension is never stored.

## 4. SavedData in 1.20.1

### Exact API — 1.20.1 has NO `SavedData.Factory`

[javap `net.minecraft.world.level.storage.DimensionDataStorage`], 1.20.1:

```
public <T extends SavedData> T computeIfAbsent(Function<CompoundTag,T>, Supplier<T>, String);
public <T extends SavedData> T get(Function<CompoundTag,T>, String);
public void set(String, SavedData);
```

[javap `net.minecraft.world.level.saveddata.SavedData`]:

```
public abstract CompoundTag save(CompoundTag);
public void setDirty();
public boolean isDirty();
```

`javap` for `SavedData$Factory` returns nothing — **the `SavedData.Factory` record (with the
`DataFixTypes` parameter) does not exist in 1.20.1**; it appears in 1.20.2+/NeoForge. NeoForge
1.20.4 docs show the newer form
(`computeIfAbsent(new Factory<>(Data::create, Data::load), "name")`) —
https://docs.neoforged.net/docs/1.20.4/datastorage/saveddata/ — do **not** copy that into 1.20.1
code. Forge 1.20.x docs (https://docs.minecraftforge.net/en/1.20.x/datastorage/saveddata/)
document the 3-arg form: "a function to load NBT data ... a supplier to construct a new instance
... and the name of the `.dat` file". `DataFixTypes` is therefore a non-issue on 1.20.1 (no
parameter to supply; vanilla applies no datafixing to modded SavedData).

The current mod's `get()` is correct for 1.20.1:
```java
server.overworld().getDataStorage().computeIfAbsent(HomesData::load, HomesData::new, DATA_NAME);
```
Storage is per-dimension (`ServerLevel#getDataStorage`); anchoring global data to
`server.overworld()` is the standard trick (file: `<world>/data/<name>.dat`).

### Codec-backed SavedData (cleaner than manual CompoundTag walking)

Use `NbtOps` inside `save`/`load` with a map codec — replaces ~50 lines of manual tag code:

```java
private static final Codec<Map<UUID, Map<String, StoredPos>>> HOMES_CODEC =
    Codec.unboundedMap(UUIDUtil.STRING_CODEC, Codec.unboundedMap(Codec.STRING, StoredPos.CODEC));
// (UUIDUtil.STRING_CODEC and UUIDUtil.CODEC (int-array) both exist in 1.20.1)

@Override
public CompoundTag save(CompoundTag tag) {
    tag.put("homes", HOMES_CODEC.encodeStart(NbtOps.INSTANCE, homes)
        .resultOrPartial(LOGGER::error).orElseGet(CompoundTag::new));
    return tag;
}
public static HomesData load(CompoundTag tag) {
    var homes = HOMES_CODEC.parse(NbtOps.INSTANCE, tag.get("homes"))
        .resultOrPartial(LOGGER::error).orElse(Map.of());
    return new HomesData(homes);   // copy into mutable maps
}
```

### Per-player alternatives, and what's idiomatic for homes

- **Capabilities** (`AttachCapabilitiesEvent<Entity>`, `ICapabilitySerializable`,
  `@AutoRegisterCapability`): docs at
  https://docs.minecraftforge.net/en/1.20.x/datastorage/capabilities/ — data attached to the
  player entity, but "the data has to be manually copied when the player entity is cloned during
  the respawn process" via `PlayerEvent$Clone` (check `#isWasDeath`; also the old entity's caps
  are invalidated — `reviveCaps()` dance). Heavy boilerplate; only pays off when other mods should
  query the data through the cap system.
- **`entity.getPersistentData()`**: ad-hoc NBT bag; also lost on death unless copied in
  `PlayerEvent.Clone`. Fine for throwaway flags, wrong for homes.
- **Central SavedData keyed by UUID** (current approach): survives death trivially, one file,
  accessible for offline players (e.g. admin listing homes of an offline player, as FTB's
  `listhomes <player>` does). **This is the idiomatic choice for homes** — keep it.
- **Flat per-player files** (FTB Essentials): SNBT files under
  `server.getWorldPath(new LevelResource("ftbessentials"))`, a `needSave` flag, flushed from a
  tick/save hook (`util/FTBEPlayerData.java`, `util/FTBEWorldData.java`). Works, but SavedData
  gives you save-on-world-save for free; don't imitate unless you need SNBT readability.

## 5. `@Mod.EventBusSubscriber` semantics & double registration

- Forge docs (https://docs.minecraftforge.net/en/1.20.x/concepts/events/): the annotation is
  "essentially equivalent to adding `MinecraftForge.EVENT_BUS.register(AnnotatedClass.class);`
  at the end of the `@Mod` class's constructor" — it registers the **class** (static handlers
  only), on the specified bus, filtered by `modid` and optional `value = Dist...`. Docs recommend
  always specifying `modid` and `bus`.
- **Double-registration question, answered from EventBus 6.0.5 bytecode**
  (`net.minecraftforge.eventbus.EventBus#register`, jar in gradle cache):

  ```
  register(Object target):
      if (listeners.containsKey(target)) return;   // <-- early-out guard
      if (target.getClass() == Class.class) registerClass((Class) target);
      else registerObject(target);
  ```

  Registering the same `Class` object twice is a **silent no-op**, so the current mod's
  constructor call `MinecraftForge.EVENT_BUS.register(CommandRegistry.class)` on a class already
  annotated `@Mod.EventBusSubscriber` does *not* double-fire handlers — it's merely redundant and
  misleading. (Registering two *different instances* of a class WOULD duplicate handlers; the
  guard is identity-per-target.)
- **Clean pattern** — pick exactly one:

  ```java
  // (a) annotation only — nothing in the mod constructor:
  @Mod.EventBusSubscriber(modid = CoreCommands.MOD_ID, bus = Mod.EventBusSubscriber.Bus.FORGE)
  public final class CommandEvents {
      @SubscribeEvent
      public static void onRegisterCommands(RegisterCommandsEvent e) {
          HomeCommands.register(e.getDispatcher());
          TpaCommands.register(e.getDispatcher());
      }
  }
  // (b) or explicit, in the @Mod constructor (what FTB Essentials Forge does):
  MinecraftForge.EVENT_BUS.addListener(this::onRegisterCommands);
  ```

  `addListener` (method reference) avoids the annotation scanner entirely and is the style used in
  `FTBEssentialsForge.java`.

## 6. Forge PermissionAPI in 1.20.1

All from [forge src `net/minecraftforge/server/permission/**`], present and unchanged in 47.2.0:

```java
// nodes/PermissionNode.java
public PermissionNode(ResourceLocation nodeName, PermissionType<T> type,
                      PermissionResolver<T> defaultResolver, PermissionDynamicContextKey... dynamics)
public PermissionNode(String modID, String nodeName, PermissionType<T> type,
                      PermissionResolver<T> defaultResolver, PermissionDynamicContextKey... dynamics)
// PermissionResolver<T>: T resolve(@Nullable ServerPlayer player, UUID playerUUID, PermissionDynamicContext<?>... context)

// nodes/PermissionTypes.java
public static final PermissionType<Boolean> BOOLEAN;   // also INTEGER, STRING, COMPONENT

// PermissionAPI.java
public static <T> T getPermission(ServerPlayer player, PermissionNode<T> node,
                                  PermissionDynamicContext<?>... context)
// also an offline-player overload getOfflinePermission(UUID, node, ctx...)
```

Rules (from the Forge javadocs in source):
- "Each node should be registered via the `PermissionGatherEvent.Nodes`" (Forge bus event, has
  `addNodes(...)`) "and stored statically in a field"; querying an unregistered node throws
  `UnregisteredPermissionException`.
- The active handler is chosen by server config (`permission.handler`); default is
  `DefaultPermissionHandler` (identifier `forge:default_handler`), which just evaluates each
  node's `defaultResolver`. Alternative handlers can be installed via
  `PermissionGatherEvent.Handler` (this is how LuckPerms-on-Forge/`luckperms` integrates).

Idiomatic dual-support pattern (common across server mods):

```java
public static final PermissionNode<Boolean> TPHERE = new PermissionNode<>(MOD_ID, "command.tphere",
    PermissionTypes.BOOLEAN, (player, uuid, ctx) -> player != null && player.hasPermission(2));

@SubscribeEvent static void onGatherNodes(PermissionGatherEvent.Nodes e) { e.addNodes(TPHERE, ...); }

// in the command tree: .requires(src -> !(src.getEntity() instanceof ServerPlayer sp)
//                                    ? src.hasPermission(2)
//                                    : PermissionAPI.getPermission(sp, TPHERE))
```

The resolver defaulting to `hasPermission(n)` means everything behaves vanilla-like (ops work)
until a permission mod overrides the node. Caveats:
- `requires()` gives a `CommandSourceStack`, PermissionAPI needs a `ServerPlayer` — branch on the
  source entity, keep `hasPermission(int)` as the console/command-block fallback.
- `.requires` results are baked into the command tree the client sees; after rank changes the
  tree isn't re-sent automatically (`server.getPlayerList().sendPlayerPermissionLevel` /
  `Commands.sendCommands` refresh it).

What real mods do: FTB Essentials skips Forge's PermissionAPI entirely and probes
FTB Ranks / LuckPerms behind its own `PermissionsProvider` interface
(`integration/PermissionsHelper.java` — `Platform.isModLoaded("luckperms")` etc.), falling back to
permission levels. For a lean mod, plain `hasPermission(2)` gating for admin commands + optional
PermissionNodes is the sweet spot. Example repo using the raw API:
https://github.com/VeiTrr/ForgePermissionApiExample

## 7. Server-only mod marking (vanilla clients can join)

Two equivalent mechanisms in 1.20.1:

1. **mods.toml** (simplest — no code):
   ```toml
   displayTest="IGNORE_SERVER_VERSION"   # "no red X if present on server but not client — use if server-only"
   ```
   (Wording straight from the MDK mods.toml template in this repo. `MATCH_VERSION` is the default;
   `IGNORE_ALL_VERSION` is for client-only cosmetic mods; `NONE` = you register your own test.)

2. **Code** (what FTB Essentials 1.20.1 Forge entrypoint does — needed only for custom logic, or
   pre-displayTest-key Forge versions):
   ```java
   ModLoadingContext.get().registerExtensionPoint(IExtensionPoint.DisplayTest.class,
       () -> new IExtensionPoint.DisplayTest(() -> NetworkConstants.IGNORESERVERONLY, (a, b) -> true));
   ```
   [javap fmlcore `IExtensionPoint$DisplayTest`]:
   `record DisplayTest(Supplier<String> suppliedVersion, BiPredicate<String,Boolean> remoteVersionTest)`
   with constant `IGNORESERVERONLY`; `NetworkConstants.IGNORESERVERONLY` aliases it [forge src].
   Source: https://github.com/FTBTeam/FTB-Essentials/blob/1.20.1/main/forge/src/main/java/dev/ftb/mods/ftbessentials/forge/FTBEssentialsForge.java
   (Note: `NetworkConstants` was removed in 1.20.4+, so guides mentioning its absence are about
   newer versions — it exists in 1.20.1.)

Related mods.toml bugs in this repo worth fixing during refactor: `loaderVersion="[36,)"` and the
forge dependency `versionRange="[36,)"` are 1.16-era leftovers from the decompile source; for
1.20.1/47.x they should be `"[47,)"`.

Docs: https://docs.minecraftforge.net/en/latest/concepts/sides/ (displayTest discussion).

## 8. ForgeConfigSpec server config

From https://docs.minecraftforge.net/en/1.20.x/misc/config/:
- `ModConfig.Type.SERVER` configs load server-side only, **live per-world in
  `<world>/serverconfig/`** (`saves/<level>/serverconfig` in singleplayer), and are synced to
  (modded) clients. `COMMON` lives in the global `config/` dir. For gameplay knobs that a server
  admin sets per world — max homes, tpa expiry, cooldowns — SERVER is correct. (Some admins
  dislike per-world files; COMMON is the pragmatic alternative. FTB Essentials uses its own SNBT
  config in `<world>` dir with FTB Library — don't imitate, that's their ecosystem.)
- Register in the mod constructor: `ModLoadingContext.get().registerConfig(ModConfig.Type.SERVER, SPEC);`
- Values (`IntValue` etc.) are lazily cached on `.get()`; safe to read once the config is loaded
  (`ModConfigEvent.Loading`/`Reloading` fire on the **mod** bus). Command handlers run long after
  load, so plain `.get()` in executes-lambdas is fine.

```java
public final class ServerConfig {
    public static final ForgeConfigSpec SPEC;
    public static final ForgeConfigSpec.IntValue MAX_HOMES;
    public static final ForgeConfigSpec.IntValue TPA_EXPIRY_SECONDS;
    public static final ForgeConfigSpec.IntValue HOME_COOLDOWN_SECONDS;
    static {
        var b = new ForgeConfigSpec.Builder();
        b.push("homes");
        MAX_HOMES = b.comment("Maximum homes per player").defineInRange("maxHomes", 5, 1, 100);
        HOME_COOLDOWN_SECONDS = b.defineInRange("cooldownSeconds", 0, 0, 3600);
        b.pop();
        b.push("tpa");
        TPA_EXPIRY_SECONDS = b.comment("Seconds before a /tpa request expires")
                              .defineInRange("expirySeconds", 60, 5, 600);
        b.pop();
        SPEC = b.build();
    }
}
```

## 9. TPA request management patterns

### FTB Essentials (1.20.1) — `command/TPACommands.java` + `FTBEEventHandler.java`

- `public record TPARequest(String id, FTBEPlayerData source, FTBEPlayerData target, boolean here, long created)`
  — the `here` flag folds `/tpa` and `/tpahere` into one code path (accept teleports
  `here ? target→source : source→target`).
- Requests live in `HashMap<String, TPARequest>` keyed by a **random 8-hex id**; `/tpaccept <id>`
  and `/tpdeny <id>` take the id, and the request message carries **clickable components**:
  ```java
  Component.literal("Accept ✔").setStyle(Style.EMPTY.applyFormat(ChatFormatting.GREEN).withBold(true)
      .withClickEvent(new ClickEvent(ClickEvent.Action.RUN_COMMAND, "/tpaccept " + request.id()))
      .withHoverEvent(new HoverEvent(HoverEvent.Action.SHOW_TEXT, Component.literal("Click to Accept"))));
  ```
  This sidesteps the "which request does bare /tpaccept take?" ambiguity entirely.
- Duplicate guard: rejects if a source→target request already exists.
- **Expiry**: a `ServerTickEvent`-driven sweep (in `FTBEEventHandler`) iterates the map and removes
  entries where `now > created + 60000L`, messaging *both* parties "TPA request expired!". The map
  is `clear()`ed on server stopped. No per-request timers, one sweep — the standard pattern.
- Cooldown/warmup is orthogonal, per-command-type `WarmupCooldownTeleporter` objects on the player
  data, ticked centrally (`tickWarmups(server)`).

### Essential Commands (Fabric) — `teleportation/`

- `TeleportRequest` holds sender/target PlayerData, `Type {TPA_TO, TPA_HERE}`, and an `ageTicks`
  counter; a singleton `TeleportManager` with a per-server-tick `tick()`:
  expiry when `ageTicks > CONFIG.TELEPORT_REQUEST_DURATION_TICKS` (messages both sides),
  a cooldown list, and a **warmup queue** (`QueuedTeleport`) that is interrupted if the player
  moves beyond a threshold or takes damage (permission-bypassable).
- Bookkeeping is **bidirectional**: target's PlayerData has an incoming-requests map keyed by
  sender UUID; sender has an `OutgoingTeleportRequests` list; `TeleportRequest.end()` removes
  itself from both. This makes `/tpcancel` O(1) instead of the current mod's linear scan over a
  single target→requester map (which also silently overwrites when two players tpa the same
  target).

### Recommended shape for the refactor

```java
public record TpaRequest(UUID requester, UUID target, boolean here, long expiresAtMillis) {}

public final class TpaManager {           // one instance, or static like FTB
    private final Map<UUID, TpaRequest> byTarget = new HashMap<>();     // latest per target
    private final Map<UUID, TpaRequest> byRequester = new HashMap<>();  // for /tpcancel
    // add/accept/deny/cancel keep both maps in sync;
    // tick(MinecraftServer): sweep expired, notify both parties (called from TickEvent.ServerTickEvent,
    //   cheap enough every tick, or gate on server.getTickCount() % 20 == 0);
    // clear on ServerStoppedEvent; remove entries on PlayerEvent.PlayerLoggedOutEvent.
}
```
Plus: clickable accept/deny in the request message, `/tpahere` via the `here` flag, and cooldowns
from config checked at request time (FTB checks the *mover's* teleporter cooldown before even
creating the request).

## Bugs/limitations in the current implementation surfaced by this research

1. Homes don't store dimension or rotation → `/home` teleports to raw coords in the *current*
   dimension (topic 3).
2. `tpaRequests` maps target→requester only: a second `/tpa` to the same target silently
   overwrites the first; `/tpcancel` linear-scans; stale entries never cleaned (only checked
   lazily via `isTpaRequestValid`) — requester never learns of expiry (topic 9).
3. Feedback via `player.sendSystemMessage` + `TextColor.fromRgb` magic ints instead of
   `sendSuccess/sendFailure` + `ChatFormatting` (topics 1, 2).
4. Redundant manual `EVENT_BUS.register` alongside `@Mod.EventBusSubscriber` (no-op, topic 5).
5. Raw-type casts on `Commands.literal` chains — decompiler artifacts (topic 1).
6. Hardcoded `MAX_HOMES_PER_PLAYER = 5` and 60s expiry → ForgeConfigSpec SERVER config (topic 8).
7. No `displayTest` → vanilla clients see a red X / may be blocked from joining (topic 7);
   mods.toml also carries wrong `loaderVersion` `[36,)`.
8. `listHomes` leaks the internal mutable map; suggestion providers ignore typed prefix.
9. `/tpaccept` teleports without firing `EntityTeleportEvent` or ensuring destination chunk load.
